## 3. 关键业务流

### 3.1 支付下单与收银台流程
1. `mapi.php` / `submit.php` 创建订单
2. `cashier.php` 展示订单与可用支付方式
3. 用户选择支付方式后跳转 `submit2.php`
4. `submit2.php` 根据商户、用户组、费率、通道配置选择插件
5. 调用 `Plugin::loadForSubmit()` 进入实际支付插件

### 3.2 支付状态查询 / 防刷流程
1. 客户端调用 `getshop.php`
2. `act=captcha_verify` 时走 `verify_captcha4()`
3. 正常应返回防护 `key`
4. 但当前实现存在**验证码失败后仍继续发 key** 的逻辑错误

### 3.3 设备付款码支付流程
1. 设备向 `gateway.php` POST JSON
2. 仅凭 `sn`（设备号 / `voice_devid`）识别商户
3. 读取 `data` 中的付款码
4. 直接创建订单并调用通道提交支付
5. **未见任何签名、时间戳、设备密钥、重放保护**

### 3.4 微信点金计划结果页
1. 访问 `gold.php?out_trade_no=...`
2. 代码按订单号查单
3. **未校验 `check_code`**
4. **未校验订单支付状态**
5. 若订单存在，直接展示“支付成功”并触发 `jumpOut`

## 1. 设备网关仅凭设备号鉴权，可被冒充发起付款码支付/恶意下单

- **风险级别**：**高危**
- **影响**：
  - 攻击者在知道或猜到有效 `voice_devid` 后，可直接冒充设备调用网关
  - 可为目标商户**创建伪造订单**
  - 若攻击者掌握有效付款码（`auth_code`），可尝试**未授权发起付款码扣款**
  - 可造成**订单污染、风控绕过、手续费/余额损耗、业务滥用**
  - 响应信息差异还可用于**设备号枚举**

### 证据位置

**文件：`gateway.php`**

关键代码：

```php
$json = file_get_contents('php://input');
$arr = json_decode($json,true);
if(!$arr || !isset($arr['cmd'])) exit;
...
}elseif($arr['cmd'] == 'qrcode'){
    $sn = $arr['sn'];
    if(!$sn) voicemsg('设备号不能为空');
    $userrow=$DB->getRow("SELECT `uid`,`gid`,`key`,`money`,`mode`,`pay`,`cert`,`status`,`channelinfo`,`qq`,`ordername`,`keytype`,`publickey` FROM `pre_user` WHERE `voice_devid`=:voice_devid LIMIT 1", [':voice_devid'=>$sn]);
    if(!$userrow) voicemsg('设备关联的商户不存在');
```

仅按 `voice_devid` 查商户，**没有签名校验**。

后续直接创建订单并提交：

```php
if(!$DB->exec("INSERT INTO `pre_order` ..."))voicemsg('创建订单失败');
...
$res = \lib\Plugin::loadForSubmit($submitData['plugin'], $trade_no, true);
```

### 漏洞成因

- 把 `voice_devid` 当成认证凭据使用
- 未见：
  - 设备密钥
  - HMAC/签名
  - 时间戳
  - nonce
  - 请求重放校验
  - 来源 IP 白名单
- 属于**认证边界设计错误**

### 利用条件

- 知道、泄露或可枚举有效的 `voice_devid`
- 商户状态正常且付款码通道可用
- 若要真实发起扣款，还需有效付款码；若仅恶意下单，使用“格式上可识别”的码即可探测系统行为

### PoC / 利用思路

#### 1）探测设备是否存在
```http
POST /gateway.php HTTP/1.1
Host: target
Content-Type: application/json

{"cmd":"qrcode","sn":"TEST-SN-001","data":"MjgxMjM0NTY3ODkwMTIzNDU2","money":"1.00","msgid":"poc001"}
```

可能出现的差异响应：

- 不存在：`设备关联的商户不存在`
- 存在但受限：`当前商户已被封禁`
- 存在且进入后续流程：返回其他业务错误/成功语音内容

#### 2）冒充设备发起订单
```http
POST /gateway.php HTTP/1.1
Host: target
Content-Type: application/json

{
  "cmd":"qrcode",
  "sn":"已知有效设备号",
  "data":"MjgxMjM0NTY3ODkwMTIzNDU2",
  "money":"10.00",
  "msgid":"evil-order-1001"
}
```

> 说明：`data` 是 `base64(auth_code)`。只要能通过 `getScanPayType()` 的类型识别，就可能继续下单并尝试提交通道。

### 修复建议

1. **不要把设备号当作认证凭据**
2. 为每个设备分配独立 `app_id/app_secret`
3. 对请求体进行 **HMAC-SHA256 签名**
4. 强制校验：
   - 时间戳
   - nonce
   - 签名
   - 请求过期时间
5. 增加 `msgid` 幂等控制，防重放
6. 对网关来源做：
   - IP 白名单 / VPN
   - mTLS（双向 TLS）优先
7. 错误信息统一化，避免枚举
8. 增加频率限制和审计告警

## 2. `gold.php` 直接回显 `out_trade_no`，存在同源反射型 XSS

- **风险级别**：**高危**
- **影响**：
  - 攻击者可构造恶意链接，诱导管理员/商户/客服访问
  - 在**主站同源**下执行任意 JS
  - 可进一步利用同源权限访问后台接口、读取页面、发起已登录请求
  - 若后台接口缺少更强的服务端校验，可演变为**后台操作劫持**

### 证据位置

**文件：`gold.php`**

```php
$out_trade_no = $_GET['out_trade_no'];
...
if($out_trade_no){
    $order = $DB->getRow("SELECT * FROM pre_order WHERE trade_no=:trade_no limit 1", [':trade_no'=>$out_trade_no]);
    ...
    if($order){
        ...
    }else{
        $errmsg = '订单号不存在<br/>out_trade_no='.$out_trade_no;
    }
}
...
<p><?php echo $errmsg?></p>
```

`$out_trade_no` 直接拼接到 HTML 中，**未做 `htmlspecialchars()`**。

### 漏洞成因

- 未对 GET 参数做输出编码
- 在 HTML 上下文中直接输出用户输入

### 利用条件

- 受害者访问攻击者构造的 URL
- 无需登录
- 无需真实订单号（走“订单不存在”分支即可触发）

### PoC / 利用思路

```text
/gold.php?out_trade_no=%3Csvg/onload=alert(document.domain)%3E
```

访问后页面会进入错误提示分支，并在：

```html
<p>订单号不存在<br/>out_trade_no=<svg/onload=alert(document.domain)></p>
```

处执行脚本。

> 由于该页面与后台同源，若管理员已登录，XSS 可被用于发起后台请求。

### 修复建议

1. 对所有输出到 HTML 的变量做编码：
   ```php
   echo htmlspecialchars($errmsg, ENT_QUOTES, 'UTF-8');
   ```
2. `out_trade_no` 应做白名单校验，只允许字母、数字、`-`、`_`
3. 错误信息不要拼接原始输入
4. 后台和支付前台建议**分离域名**
5. 增加 CSP（Content-Security-Policy）降低 XSS 利用成功率

## 3. `cashier.php` 使用可控 `sitename` 直接输出，存在反射型 XSS

- **风险级别**：**中危**
- **影响**：
  - 攻击者可在收银台页面注入 JS
  - 可用于钓鱼、篡改支付页面、窃取同源信息、发起后台同源请求
  - 该页面面向付款用户，风险面广

### 证据位置

**文件：`cashier.php`**

```php
$sitename=base64_decode(daddslashes($_GET['sitename']));
...
<title>收银台 | <?php echo $sitename?$sitename:$conf['sitename']?> </title>
...
<p> <?php echo $sitename?$sitename:$conf['sitename']?></p>
```

其中 `sitename` 来自 GET 参数，经 `base64_decode()` 后直接输出，**未转义**。

### 漏洞成因

- 把 Base64 当作“安全处理”，但 Base64 只是编码，不是过滤
- 输出阶段未做 HTML 编码

### 利用条件

- 需要一个**存在的 `trade_no`**
- 受害者访问恶意构造的收银台链接

### PoC / 利用思路

假设存在有效订单号 `20251110123456789`：

```text
/cashier.php?trade_no=20251110123456789&sitename=PHN2Zy9vbmxvYWQ9YWxlcnQoZG9jdW1lbnQuZG9tYWluKT4=
```

其中 Base64 解码后为：

```html
<svg/onload=alert(document.domain)>
```

将被输出到页面标题/页脚等位置。

### 修复建议

1. 对 `sitename` 做白名单校验（长度、字符集）
2. 输出时统一：
   ```php
   echo htmlspecialchars($sitename, ENT_QUOTES, 'UTF-8');
   ```
3. 不要允许前端通过 GET 参数覆盖页面站点名
4. 建议仅允许服务端内部配置项控制页面标题

## 4. `getshop.php` 验证码失败后仍继续签发 key，可绕过验证码前置校验

- **风险级别**：**中危**
- **影响**：
  - 攻击者可在验证码失败的情况下，仍拿到后续流程所需 `key`
  - 导致验证码/风控前置校验**形同虚设**
  - 可能被用于自动化刷单、脚本化轮询、支付防护绕过

### 证据位置

**文件：`getshop.php`**

```php
case 'captcha_verify':
    $pid=$_POST['pid'];
    $trade_no=$_POST['trade_no'];
    if(!$pid || !$trade_no)exit(json_encode(['code'=>-1, 'msg'=>'参数不完整']));
    $captcha_result = verify_captcha4();
    if($captcha_result !== true){
        echo json_encode(['code'=>-1, 'msg'=>'验证失败，请重新验证']);
    }
    $key = time().getDefendKey($pid, $trade_no).rand(111111,999999);
    echo json_encode(['code'=>0, 'key'=>$key]);
break;
```

验证码失败后只 `echo` 错误，**没有 `exit` / `return`**，会继续执行并发放 `key`。

### 漏洞成因

- 典型控制流错误
- 校验失败后没有中止执行

### 利用条件

- 可访问 `getshop.php`
- 知道 `pid` 和 `trade_no`
- `checkRefererHost()` 若只是检查 Referer，则**不是安全边界**，服务端请求可伪造

### PoC / 利用思路

```bash
curl 'https://target/getshop.php?act=captcha_verify' \
  -H 'Referer: https://target/' \
  -d 'pid=1000&trade_no=20251110123456'
```

若验证码参数缺失/错误，仍可能得到类似响应：

```json
{"code":-1,"msg":"验证失败，请重新验证"}{"code":0,"key":"1731234567xxxxxxxx123456"}
```

攻击者可直接提取第二段 JSON 中的 `key`。

### 修复建议

1. 校验失败后立即终止：
   ```php
   if($captcha_result !== true){
       exit(json_encode(['code'=>-1, 'msg'=>'验证失败，请重新验证']));
   }
   ```
2. `key` 不应直接裸返回给未绑定会话的请求
3. `key` 应与：
   - 服务端 session
   - 已验证的 captcha token
   - 订单号
   - 过期时间
   强绑定
4. 不要将 Referer 校验视为安全控制

## 5. `gold.php` 未校验 `check_code` 且未检查订单支付状态，可伪造“支付成功”页面

- **风险级别**：**中危**
- **影响**：
  - 任何人只要知道一个订单号，就可能访问到“支付成功”界面
  - 页面会自动向微信父页面发送 `jumpOut`
  - 会误导用户/客服/商家认为已支付
  - **资金状态本身不会被代码改为已支付**，但若下游业务错误地信任同步跳转结果，可能引发业务损失

### 证据位置

**文件：`gold.php`**

读取但未使用：

```php
$sub_mch_id = $_GET['sub_mch_id'];
$out_trade_no = $_GET['out_trade_no'];
$check_code = $_GET['check_code'];
```

查询订单后，**未校验支付状态**：

```php
if($order){
    $trade_no = $order['trade_no'];
    $jump_url = $siteurl.'pay/return/'.$trade_no.'/';
}
```

后续直接展示成功页面：

```php
<div class="text">¥<?php echo $order['money']?></div>
<p>支付成功！请点击按钮返回商家页面</p>
...
mchData = {action:'jumpOut', jumpOutUrl:'<?php echo $jump_url?>'};
```

### 漏洞成因

- `check_code` 完全未参与校验
- 未验证 `$order['status']`
- 直接把“订单存在”当作“支付成功”

### 利用条件

- 知道一个存在的订单号
- 不要求订单已支付

### PoC / 利用思路

```text
/gold.php?out_trade_no=<存在但未支付的订单号>
```

页面仍会显示：

- 金额
- “支付成功”
- 自动 `jumpOut`

> 是否能进一步导致商家错误放货，取决于 `pay/return/<trade_no>/` 以及商家自身业务是否再次校验支付状态。该部分源码未提供，需继续人工复核。

### 修复建议

1. 按微信官方文档校验 `check_code`
2. 校验 `sub_mch_id` 与订单所属商户一致
3. 仅当订单状态明确为已支付时才显示成功
4. 未支付状态应展示“处理中/未支付”，禁止 `jumpOut`
5. 不允许前端展示结果替代服务端支付状态判断

$doc = trim($_GET['doc']);

$s = isset($_GET['s'])?$_GET['s']:exit('404 Not Found');

- 路由参数 `mod` / `doc` / `s` 直接进入加载器

- 加载器若使用拼路径或动态 `include`

## 7. 待验证风险：多处核心查询依赖字符串拼接，需确认 `daddslashes()` 是否足够安全

- **风险级别**：**高危（待验证）**
- **影响**：
  - 若 `daddslashes()` 实现不可靠，可能导致 SQL 注入
  - 涉及订单查询等核心业务流，影响面大

### 证据位置

### 漏洞成因

- 核心查询仍存在字符串拼接

  - 考虑 `NO_BACKSLASH_ESCAPES`
  - 处理多字节绕过
  - 完整覆盖边界场景

### 利用条件

- `daddslashes()` 存在实现缺陷
- DB 层未额外使用参数化处理

### PoC / 利用思路

> 若被 `daddslashes()` 正确拦截，则无效；否则将存在高危 SQLi。

1. 全面改用预编译参数化查询
2. 禁止业务层拼接 SQL
3. 对现有 DB helper 做统一治理和审计

# 五、整体结论

## 1. 总体评价

基于当前快照，可确认该项目在**公共入口**上存在多处真实安全问题，尤其是：

- **设备付款网关认证缺失**
- **同源反射型 XSS**
- **验证码逻辑缺陷**
- **支付结果页状态校验缺失**

这些问题都位于**无需登录即可访问**的入口，风险较高、利用门槛较低。

同时，由于本次快照未覆盖多个核心实现文件，仍存在以下高价值待查方向：

- 路由动态加载是否可导致 LFI
- `daddslashes()` 是否可被绕过
- 下单接口是否允许注入恶意订单名
- 管理/商户 AJAX 接口是否存在 CSRF / 越权

---

## 2. 优先修复顺序

### 第一优先级（立即修复）
