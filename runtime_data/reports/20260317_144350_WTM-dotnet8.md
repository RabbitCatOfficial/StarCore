### 1. `_remotetoken` 未验签导致会话伪造 / 登录绕过

- **风险级别：严重**
- **影响：**
  - 在 `HasMainHost=false` 场景下，攻击者可伪造 `_remotetoken`，借助缓存中的 `LoginUserInfo` 直接完成登录。
  - 若目标用户 7 天内登录过，且缓存未被清理，即使已退出登录，仍可能被接管会话。
  - 可直接接管后台、调用受保护 API、读取/修改业务数据。

#### 证据位置
- `src/WalkingTec.Mvvm.Core/WTMContext.cs:181-206`
  - 读取查询参数 `_remotetoken`
  - 仅通过 `new JwtSecurityToken(remoteToken)` 解析 token
  - 未看到签名、过期、issuer、audience 校验
- `src/WalkingTec.Mvvm.Core/WTMContext.cs:189`
  - `token = new JwtSecurityToken(remoteToken);`
- `src/WalkingTec.Mvvm.Core/WTMContext.cs:196, 201, 206`
  - 直接按 `itcode/tenant` 查缓存或 `ReloadUser(usercode)`
- `src/WalkingTec.Mvvm.Core/Extensions/SystemExtensions/DistributedCacheExtensions.cs:117-120`
  - `LoginUserInfo` 默认缓存 **7 天**
- `src/WalkingTec.Mvvm.Core/WTMContext.cs:238`
  - 只有显式 `LoginUserInfo = null` 才删缓存
- `demo/WalkingTec.Mvvm.Demo/Controllers/LoginController.cs:122-125`
- `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/AccountController.cs:228,237-238`
  - 退出登录仅清 Session/SignOut，**未清用户缓存**
- `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/AccountController.cs:80-89`
  - `LoginRemote` 为公开接口，且在 `Wtm.LoginUserInfo != null` 时直接 `SignInAsync`
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:719-731`
  - `RemoteEntry` 为公开接口，也会直接签入
- `src/WalkingTec.Mvvm.Core/Support/AuthConstants.cs:14,140`
  - JWT 中用户 claim 为 `itcode`，租户 claim 为 `tenant`
- `demo/WalkingTec.Mvvm.Demo/appsettings.json:127-128`
  - `mainhost` 地址未实际配置，满足 `HasMainHost=false`

#### 漏洞成因
`WTMContext.LoginUserInfo` 在查询串存在 `_remotetoken` 时，直接解析 JWT 文本并提取 `itcode/tenant`，随后使用这些值命中缓存或重载用户信息。  
这里使用的是 **“解析”**，不是 **“验证”**。

#### 利用条件
- 应用运行在 `HasMainHost=false` 分支
- 目标用户缓存存在
  - 缓存默认 7 天
  - 退出登录不清缓存
- 攻击者知道或猜到用户 `itcode`，如 demo 中默认 `admin`

#### PoC / 利用思路
**PoC 1：伪造 `_remotetoken` 直接登录**
```http
GET /api/_account/LoginRemote?_remotetoken=eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJpdGNvZGUiOiJhZG1pbiJ9.x HTTP/1.1
Host: target
```

或：

```http
GET /_Framework/RemoteEntry?redirect=/&_remotetoken=eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJpdGNvZGUiOiJhZG1pbiJ9.x HTTP/1.1
Host: target
```

说明：
- payload 中只需带 `itcode=admin`
- 此 token **无需有效签名**，因为这里只做解析不做验签
- 若缓存中存在 `admin` 用户信息，接口将直接签发 Cookie 会话

#### 修复建议
1. **禁止**通过查询参数 `_remotetoken` 直接恢复登录态。
2. 若必须保留：
   - 必须走标准 JWT 验签流程
   - 校验签名、过期、issuer、audience、算法白名单
3. `LoginRemote/RemoteEntry` 不应仅凭 `Wtm.LoginUserInfo != null` 就签入。
4. 退出登录时必须同步清理用户缓存。
5. 对该能力增加审计日志与来源限制，默认关闭。

### 2. `_FrameworkController` 动态 VM 接口可被任意登录用户滥用，造成越权读写与权限绕过

- **风险级别：高危**
- **影响：**
  - 任意已登录用户可调用 `/_Framework/*` 通用接口，读取任意 ViewModel 数据、导出数据，甚至修改任意实体字段。
  - 在 demo 中由于存在匿名注册，攻击者可先注册普通用户，再利用该接口读取后台用户数据并修改敏感字段，完成提权。

#### 证据位置
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:34`
  - 整个控制器标记为 `[AllRights]`
- `src/WalkingTec.Mvvm.Mvc/Filters/PrivilegeFilter.cs:217`
  - 只有 `isAllRights == false` 才做细粒度权限检查
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:131-143`
  - `GetPagingData(string _DONOT_USE_VMNAME, string _DONOT_USE_CS)`
  - 直接 `Wtm.CreateVM(_DONOT_USE_VMNAME, ...)`
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:180`
  - 直接返回 `listVM.GetDataJson()`
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:201-210`
  - `UpdateModelProperty(...)`
  - `vm.Entity.SetPropertyValue(field, value);`
  - `DC.SaveChanges();`
- `src/WalkingTec.Mvvm.Core/BaseVM.cs:107,111`
  - `VMFullName` 使用 `AssemblyQualifiedName`
- `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/AccountController.cs:101-127`
  - 注册接口匿名开放，且新用户默认赋予普通角色 `002`
- `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:17,20`
  - 其他敏感接口也仅要求“已登录”

#### 漏洞成因
`_FrameworkController` 作为框架级通用控制器，本应仅供前端已授权页面内部调用，但实际：
1. 整个控制器使用 `[AllRights]`，绕过细粒度功能权限；
2. `_DONOT_USE_VMNAME` 完全由客户端控制；
3. `UpdateModelProperty` 允许客户端直接指定 `field/value` 并落库；
4. 未见针对目标 VM、目标字段、目标实体的授权与白名单校验。

#### 利用条件
- 任意已登录用户
- 知道目标 VM 的 `VMFullName`
  - 通常可从正常页面请求、前端源码或 `VMFullName` 字段拿到

#### PoC / 利用思路
**场景：普通注册用户读取后台用户列表并重置 admin 密码**

**步骤 1：注册普通用户**
```http
POST /api/_account/Reg
```

**步骤 2：登录普通用户，拿 Cookie**

**步骤 3：读取后台用户列表**
```http
POST /_Framework/GetPagingData
Content-Type: application/x-www-form-urlencoded

_DONOT_USE_VMNAME=<FrameworkUserListVM 的 VMFullName>
```

返回 JSON 中可获取后台用户 `ID`。

**步骤 4：修改 admin 密码为已知 MD5**
```http
POST /_Framework/UpdateModelProperty
Content-Type: application/x-www-form-urlencoded

_DONOT_USE_VMNAME=<FrameworkUserVM 的 VMFullName>&id=<admin-guid>&field=Password&value=E10ADC3949BA59ABBE56E057F20F883E
```

随后可直接使用：
- 账号：`admin`
- 密码：`123456`

#### 修复建议
1. **移除** `_FrameworkController` 上的 `[AllRights]`，改为逐接口授权。
2. `GetPagingData/GetExportExcel/UpdateModelProperty` 必须校验：
   - 当前用户是否有目标业务页面权限
   - 当前用户是否可访问目标 VM/实体
   - 当前字段是否允许前端修改
3. 禁止客户端直接传递任意 `VMFullName`，改为服务端固定映射白名单。
4. `UpdateModelProperty` 必须做字段级白名单与业务校验，且重新执行 ModelValidation。
5. 对此类通用接口增加审计日志、限流和异常告警。

### 3. 公共 `redirect` 参数未转义，存在反射型 XSS，并伴随开放重定向

- **风险级别：高危**
- **影响：**
  - 攻击者可构造链接，在受害者浏览器中执行同源脚本。
  - 一旦管理员访问，可直接调用后台接口、发起敏感操作。
  - 即使不打 XSS，也存在未校验的外部跳转。

#### 证据位置
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:699-708`
  - `SetLanguageForBlazor(string culture, string redirect)`
  - 直接输出：`window.location.href='{HttpUtility.UrlDecode(redirect)}';`
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:719-731`
  - `RemoteEntry(string redirect)`
  - 同样直接拼入 JS 字符串
- `src/WalkingTec.Mvvm.Mvc/_FrameworkController.cs:699,719`
  - 两个接口均为 `[Public]`
- 同类未校验跳转：
  - `demo/WalkingTec.Mvvm.Demo/Controllers/LoginController.cs:28,81`

#### 漏洞成因
用户输入的 `redirect` 经 `UrlDecode` 后，未做 JS 字符串转义或本地 URL 校验，直接拼进 `<script>`。

#### 利用条件
- 无需登录
- 仅需诱导受害者访问恶意链接

#### PoC / 利用思路
```http
GET /_Framework/RemoteEntry?redirect=%27%3Balert(document.domain)%2F%2F
```

或：

```http
GET /_Framework/SetLanguageForBlazor?culture=zh-CN&redirect=%27%3Bfetch(%27/api/_account/checkuserinfo%27)%3B%2F%2F
```

响应会变成类似：
```html
<script>window.location.href='';alert(document.domain)//'</script>
```

#### 修复建议
1. **禁止**把用户输入直接拼进 `<script>`.
2. 跳转使用服务端安全 API：
   - `LocalRedirect`
   - `Url.IsLocalUrl`
3. 若确需输出到 JS：
   - 进行严格 JS/HTML 编码
4. 对 `redirect` 仅允许站内相对路径，不允许协议头、双斜杠、引号等危险字符。
5. 修复同类问题，包括登录后的 `ReturnUrl` 处理。

### 4. 文件上传 `subdir` 可控，存在路径穿越与任意目录写入（文件名随机）

- **风险级别：高危**
- **影响：**
  - 已登录用户可把文件写入上传目录之外的任意可写位置。
  - 虽然文件名被随机化，但扩展名可控，仍可用于：
    - 写入 Web 根目录下的恶意 HTML/SVG/JS
    - 存储型 XSS / 恶意文件投递
    - 覆盖业务目录结构外的可写文件夹
- 在 demo 中，匿名用户可先注册普通账号后完成利用。

#### 证据位置
- 上传入口：
  - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:26-29`
  - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:35-60`
  - `subdir/groupName/csName` 均由请求控制
- 权限：
  - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:17,20`
  - 仅要求已登录 + `[AllRights]`
- 注册普通用户：
  - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/AccountController.cs:101-127`
- 本地落盘：
  - `src/WalkingTec.Mvvm.Core/Support/FileHandlers/WtmLocalFileHandler.cs:46`
    - `pathHeader = Path.Combine(pathHeader, subdir);`
  - `src/WalkingTec.Mvvm.Core/Support/FileHandlers/WtmLocalFileHandler.cs:89-100`
    - `Path.GetFullPath(rv);`
    - 未校验是否仍在允许根目录内
  - `src/WalkingTec.Mvvm.Core/Support/FileHandlers/WtmLocalFileHandler.cs:69`
    - `File.Create(fullPath)`
- 文件名过滤极弱：
  - `src/WalkingTec.Mvvm.Core/Support/FileHandlers/WtmFileProvider.cs:100`
    - 仅去掉 `<`、`>`、空格，未限制扩展名
- demo 上传目录：
  - `demo/WalkingTec.Mvvm.Demo/appsettings.json:81,86,90`

#### 漏洞成因
`subdir` 被直接参与 `Path.Combine`，随后 `GetFullPath` 归一化，但**没有做目录边界检查**。  
因此：
- `..\..\` 可逃逸上传根目录
- 绝对路径可直接覆盖 `Path.Combine` 的前缀

#### 利用条件
- 任意已登录用户
- demo 中可先匿名注册再登录

#### PoC / 利用思路
**步骤 1：注册并登录普通用户**

**步骤 2：上传恶意 SVG 到 Web 根外的指定目录**
```http
POST /api/_file/Upload?subdir=..\\..\\wwwroot\\poc
Content-Type: multipart/form-data

file=@poc.svg;filename=poc.svg
```

服务器将返回：
```json
{"Id":"<file-guid>","Name":"poc.svg"}
```

**步骤 3：查询实际路径**
```http
GET /api/_file/GetFileInfo/<file-guid>
```

返回对象中包含 `Path/SaveMode/...`，可进一步确认写入结果。

#### 修复建议
1. 对 `subdir` 做白名单约束，禁止客户端任意传目录。
2. 归一化后校验目标路径必须位于允许根目录内。
3. 禁止绝对路径、`..`、盘符、UNC 路径。
4. 对上传扩展名和 MIME 做双重校验。
5. 将上传与静态可执行/可渲染目录隔离。
6. 对富文本上传、图片上传与普通文件上传分离策略。

3. 删除仓库中的硬编码 JWT 密钥，改走密钥管理系统。
4. 密码存储改用 `PBKDF2 / bcrypt / scrypt / Argon2`。
5. 禁止 `IsQuickDebug=true` 的配置进入生产。

  - `GetFileInfo` 还会返回内部 `Path/SaveMode/TenantCode/ExtraInfo` 等信息。

#### 证据位置

  - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:74-85`
    - `GetFileName/GetFile`
  - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:136-165`
    - `GetFileInfo/GetUserPhoto/DownloadFile`
- `GetFileInfo` 返回完整对象：
  - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:137,144`
- 文件查询仅按 ID：
  - `src/WalkingTec.Mvvm.Core/Support/FileHandlers/WtmFileProvider.cs:137-144`
  - `src/WalkingTec.Mvvm.Core/Support/FileHandlers/WtmFileProvider.cs:205-212`
- 泄露字段：

#### 漏洞成因
文件访问控制仅基于 `FileAttachment.ID`，未绑定：
- 上传用户
- 业务实体归属
- 访问者权限
- 签名 URL / 时效令牌

#### 利用条件
- 需要知道目标文件 GUID
- GUID 可能来自：
  - 上传响应
  - 业务实体字段
  - 前端源码/接口包

#### PoC / 利用思路
**用户 A 上传文件后得到 `Id`：**
```json
{"Id":"4f3c...","Name":"secret.pdf"}
```

**匿名访问：**
```http
GET /api/_file/DownloadFile/4f3c...
GET /api/_file/GetFileInfo/4f3c...
```

#### 修复建议
1. 文件下载必须走鉴权。
2. 文件表增加所有者/业务归属校验。
3. 对外下载改用一次性签名 URL。
4. `GetFileInfo` 不应返回内部 `Path/SaveMode/TenantCode` 等敏感字段。
5. 默认关闭文件公开访问。

## 三、待验证风险

### 1. 用户可控 `DONOTUSECSName/csName` 可能导致跨库访问 / 越权切库

- **标记：待验证风险**
- **证据位置**
  - `src/WalkingTec.Mvvm.Mvc/Filters/DataContextFilter.cs:57,68-73`
    - 从请求中读取 `DONOTUSECSName`
  - `src/WalkingTec.Mvvm.Core/Utils.cs:607,614,625`
    - 允许按连接名选择连接
  - `src/WalkingTec.Mvvm.Core/WTMContext.cs:682,722`
    - 根据 `cskey` 创建对应 `DataContext`
  - 文件接口也允许传 `csName`
    - `demo/WalkingTec.Mvvm.Demo/Areas/_Admin/ApiControllers/FileApiController.cs:26,29,35,60,75,83,163,182`

#### 风险说明
如果部署环境存在多个已配置连接，且不同连接间数据敏感级别不同，那么客户端可控连接名可能造成跨库读写或跨租户访问。

#### 为什么标记待验证
当前 demo 默认只显式启用 `default`，其他连接大多注释；是否实际可跨库，取决于部署时的连接配置。

#### 利用思路
```http
GET /_Framework/GetFile?id=<id>&_DONOT_USE_CS=otherdb
GET /api/_file/GetFile/<id>?csName=otherdb
```

#### 建议
- 禁止客户端直接决定连接名
- 连接选择只能由服务端租户/路由/策略决定
- 对所有 `csName/_DONOT_USE_CS/DONOTUSECSName` 做白名单或移除

### 2. `DataPrivilege` 机制疑似未覆盖普通 CRUD / List 查询链路

- **标记：待验证风险**
- **证据位置**
  - 已启用数据权限：
    - `demo/WalkingTec.Mvvm.Vue3Demo/Startup.cs:130`
  - 全局查询过滤仅处理 `IsValid/TenantCode`：
    - `src/WalkingTec.Mvvm.Core/DataContext.cs:107,112,123`
  - 数据权限拼接函数 `AppendSelfDPWhere` 的调用点很少：
    - `src/WalkingTec.Mvvm.Core/Extensions/DCExtension.cs:58,212`
  - 普通 CRUD/List 仍直接查 `DC.Set<T>()`：
    - `src/WalkingTec.Mvvm.Core/BaseCRUDVM.cs:332`
    - `demo/WalkingTec.Mvvm.Vue3Demo/Areas/SchoolData/ViewModels/CityVMs/CityListVM.cs:29`
    - `demo/WalkingTec.Mvvm.Demo/Controllers/CityApiController.cs:38,69,91`

#### 风险说明
从代码看，数据权限 where 似乎主要用于下拉框 helper，而不是普通列表/详情/编辑查询。若属实，则启用了 `DataPrivilegeInfo<T>` 的模型仍可能被普通用户越权读取或修改。

#### 为什么标记待验证
仍需用“受限账号 + 实际数据权限规则”做运行态验证，以排除其他未在当前快照中体现的拦截逻辑。

#### 建议
- 在 `BaseCRUDVM.GetById`、`BasePagedListVM` 查询入口统一强制拼接数据权限
- 将数据权限做成框架级统一拦截，而不是零散 helper 调用

## 四、整体结论

### 1. 总体判断
该仓库存在**多处框架级高风险问题**，其中最严重的是：

1. **`_remotetoken` 未验签会话伪造**
2. **`_FrameworkController` 动态 VM 通用接口权限绕过**
3. **公共 `redirect` 反射型 XSS**
4. **文件上传路径穿越导致任意目录写入**

另外，Demo 应用默认配置也明显不适合直接上线：
- 默认管理员 `admin/000000`
- 新租户默认 `admin/000000`
- JWT 密钥硬编码
- `IsQuickDebug=true`

### 2. 优先修复顺序
1. **立即下线/封禁** `_remotetoken` 登录恢复链路
2. **立即收口** `/_Framework/*` 动态 VM 接口，移除 `[AllRights]`
3. 修复 `redirect` 反射型 XSS / 开放重定向
4. 修复上传 `subdir` 路径穿越与扩展名缺失校验
5. 替换 Demo 默认密钥、默认账号、弱密码哈希
6. 给文件访问增加所有者/业务归属校验
7. 人工复核多连接与数据权限链路

### 3. 仍需人工复核的关键链路
1. **多数据库/读写分离部署**
   - 需要确认生产是否存在多个连接名，以及是否被用户可控参数影响
2. **数据权限实际生效范围**
   - 需要用真实受限账号验证列表、详情、编辑、导出
3. **所有 `_Framework/*` 的前端调用面**
   - 尤其是 `_DONOT_USE_VMNAME`、`_DONOT_USE_CS` 的实际暴露情况
4. **文件渲染路径与浏览器行为**
   - 尤其是 HTML/SVG 在真实部署中的可执行性

### 4. 本次未发现的高危类型
在当前审计范围内，**未发现证据充分的**：
- 直接 SQL 注入
- 直接命令执行
- 明确可利用的反序列化 RCE

但这不代表完全不存在，建议结合真实部署配置继续验证。
